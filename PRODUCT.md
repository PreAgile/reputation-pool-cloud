# reputation-pool-cloud — 제품 문서

> 이 레포는 **비공개 상용부**다. 공개 OSS [reputation-pool](https://github.com/PreAgile/reputation-pool)
> 위에 올리는 호스티드 SaaS의 모든 것(서버 껍데기, 테넌시, 과금, 프론트, 배포)이 여기 산다.
> 문서·이슈·커밋은 한글. 코드 식별자는 영어.

## 한 줄 비전

"리소스(프록시/계정/세션) 풀의 평판 관리를 API 하나로" — 스크래핑·자동화 인프라를 운영하는 팀이
직접 만들던 쿨다운/블록리스트/리스 로직을, 검증된 엔진의 호스티드 API로 대체한다.

## open-core 경계 규칙 (가장 중요한 규칙)

| 어디로 | 무엇이 |
|---|---|
| **공개 reputation-pool** (영어) | 엔진을 더 좋게 만드는 모든 것 — 도메인 로직, 정책, 포트, 성능, 테스트 |
| **이 레포** (한글) | 호스팅/판매에 관한 모든 것 — 테넌시, 인증, 과금, 미터링, 프론트, 배포, 운영 |

- 이 레포는 공개 레포를 **fork하지 않는다**. Maven Central의 `reputation-pool-core` 아티팩트를 의존성으로 소비한다.
- dogfooding 중 나오는 엔진 개선 요구는 이 레포의 이슈가 아니라 **공개 레포의 이슈**로 낸다.
- 이 레포 전체를 나중에 오픈소스로 여는 시나리오는 없다. 공개할 가치가 생긴 코드는 공개 레포로 **추출**한다.

## 타깃 고객 가설 (검증 전)

1. 스크래핑/데이터 수집 인프라 팀 — 프록시 풀 회전·쿨다운을 손수 구현 중인 곳
2. 계정 기반 자동화 운영자 — 계정 밴/쿨다운 관리가 매출에 직결되는 곳
3. 프록시 리셀러 — 자기 상품에 "평판 관리 내장"을 부가가치로 붙이고 싶은 곳

가격 가설, 시장 크기, 경쟁(직접 구현 외 대안이 있나)은 Phase 1 dogfooding 후 업데이트.

## 아키텍처 스케치

```
[고객] ──REST/gRPC──▶ [Spring Boot 셸]  ← 이 레포
                        ├─ API key 인증 · 테넌트 라우팅 · 미터링
                        ├─ reputation-pool-core 0.2.0 (Maven Central 의존)
                        ├─ reputation-pool advisor 패턴 재사용 (gRPC or 내장)
                        └─ PostgreSQL (스냅샷 + 감사, 공개 레포의 persistence 스키마)
[운영자/고객] ──▶ [대시보드 프론트] — 풀 상태 · 평판 곡선 · 이벤트 스트림
배포: private 서버, Docker Compose로 시작 (k8s는 필요해질 때)
```

## 로드맵

**전략 (2026-07-15 확정): 프로덕션 레벨로 한 번에 완성 후 검증.** 단계적 dogfood MVP가 아니라,
판매 가능한 멀티테넌트 SaaS(10개 도메인: 백엔드·테넌시·API·인증·대시보드·과금·관측·인프라·온보딩·법무)를
완성하고 종료 기준 5개를 통과한 뒤 core 1.0.0을 선언한다.
마스터 플랜: [docs/production-roadmap.html](docs/production-roadmap.html) (빌드 순서·결정 대기·검증 기준의 원본).

빌드 단계 = 마일스톤 4개 (각 단계의 세부 작업은 해당 마일스톤의 이슈로 관리):

1. [`v1-backend`](../../milestones/1) — 엔진이 gRPC 뒤에서 실제로 뜨는 토대 (#1–#3, #5)
2. [`v1-tenancy`](../../milestones/2) — 멀티테넌시 + 인증/키 + 미터링 수집 (#4, #9, #10)
3. [`v1-dashboard`](../../milestones/3) — REST 컨트롤 플레인 + 대시보드 UI (#11, #12)
4. [`v1-launch`](../../milestones/4) — 청구·모니터링·인프라·온보딩·법무·검증 (#6, #7, #13–#16)

선행 게이트(불변): 공개 레포 core+persistence **0.2.0 Maven Central 배포** — 그 전엔 백엔드가 컴파일 불가.
(초기 Phase 1 dogfood 설계는 [docs/phase1-plan.html](docs/phase1-plan.html)에 기록으로 보존 — D1 백엔드 토대 부분은 그대로 유효.)

## 결정 대기 목록

- [ ] 고객 노출 API: gRPC 그대로 vs REST 게이트웨이 (고객 온보딩 마찰 vs 구현 비용)
- [ ] 테넌시 시작점: 테넌트당 컨테이너(격리 쉬움, 운영 무거움) vs 공유 프로세스 멀티테넌트(반대)
- [ ] 프론트 스택 (Phase 2에서)
- [ ] 가격 모델 가설 (리스 횟수? 풀 크기? 좌석?)

## 이 레포가 절대 하지 않는 것

- 공개 레포 코드 복사/포크
- 엔진 도메인 로직 재구현 (전부 core 의존으로)
- 공개 레포에 상용 계획 노출

REPUTATION-POOL-CLOUD · PHASE 1 (DOGFOOD MVP) · 설계 결정

엔진은 이미 있다 —
SaaS Phase 1은 "껍데기·테넌시·판매"만 짓는다

공개 레포가 무엇을 완성해 뒀는지 재고 조사한 뒤, cloud가 시작하기 전에 못 박아야 할 근본 결정 3개와 Phase 1에 남은 실제 작업 목록. "SaaS에 뭐가 남았지?"에 대한 답.

결론 먼저

Phase 1의 엔진은 core + persistence를 Maven Central에서 라이브러리로 소비(→ 0.2.0에 persistence도 배포하는 결정이 선행), gRPC 서버(AdvisorServer)는 통째로 임베드하지 않고 그 composition-root 레시피를 Spring Boot 구성으로 이식, 고객 노출은 Phase 1은 gRPC 그대로(dogfood는 내가 클라이언트 — REST 게이트웨이는 외부 고객이 생기는 Phase 3), 테넌시는 싱글테넌트로 시작하되 "테넌트 = 풀 네임스페이스" 공유 프로세스 방향으로 설계. 남은 실제 작업은 §3의 7개 — 전부 "호스팅/운영/판매" 층이고 엔진 로직은 한 줄도 새로 안 짠다.

§1재고 조사 — 공개 레포가 이미 준 것 vs cloud가 지어야 할 것

코드로 확인: Maven Central엔 core만, 하지만 server 모듈에 완성된 참조 배선이 있다

✓ 이미 있음 (재사용)
  • 엔진: ResourcePool·ReputationEngine — Lincheck으로 선형성 증명됨
  • 영속: PostgresResourceStore(스냅샷)·PostgresAuditTrail(감사+보존정책)
  • gRPC 계약: advisor.proto — 6개 RPC, buf로 하위호환 가드됨
  • 참조 배선: AdvisorServer가 복원·주기 체크포인트·감사 fan-out·셧다운 저장을 이미 조립 — 레시피로 그대로 참고
◆ cloud가 지어야 함 (Phase 1)
  • Spring Boot 셸 (라이프사이클·설정·헬스체크)
  • API key 인증 (싱글 키)
  • 테넌트 개념 (Phase 1은 1개)
  • Docker Compose 배포 (앱 + PostgreSQL)
  • 미터링·과금 훅 자리 (Phase 3 전까지 비움)
  • 대시보드 (Phase 2)
핵심 통찰: cloud는 엔진을 새로 만들 필요가 전혀 없다. AdvisorServer가 "core+persistence를 어떻게 배선하는가"의 완성된 답을 이미 갖고 있어서, cloud의 일은 그 배선을 Spring Boot 방식으로 옮기고 그 위에 테넌시·인증·과금을 두르는 것뿐. 위험한 로직(동시성·복원·보존)은 전부 검증된 라이브러리 안에 있다.

§2못 박아야 할 근본 결정 3개

Phase 1 코드 첫 줄 전에 정해야 하는 것 — 뒤집으면 비싼 것들

결정 1엔진을 어떻게 손에 넣나 — persistence 배포가 열쇠
  • core만 + cloud 자체 스토어 구현: Maven Central 현재 그대로. 단, 잘 만들어 검증된 PostgresResourceStore를 버리고 ResourceStore·EventSink 포트를 cloud가 다시 구현 — 낭비 + open-core 경계 위반(엔진 주변부를 cloud에 복제)
  • core + persistence 둘 다 Maven Central 배포: persistence에 publish 설정 추가(0.2.0 범위). cloud는 둘 다 의존만 하면 됨. persistence가 공개 라이브러리가 되지만, 애초에 "데모 어댑터"로 공개돼 있던 코드라 손해 없음 — 오히려 "쓸 수 있는 Postgres 어댑터"가 OSS 가치를 더함
  • 소스 의존/멀티레포 서브모듈: 빌드 복잡, open-core 경계 흐림
→ persistence도 0.2.0에 배포. #55 릴리스 범위를 "core만"에서 "core + persistence"로 넓히는 작은 결정이 Phase 1을 크게 단순화한다. server 모듈은 배포 불필요(참조 코드로만).
결정 2AdvisorServer를 임베드하나, 레시피만 베끼나
  • 통째로 임베드: server 모듈도 배포하고 AdvisorServer.create(...)를 Spring 빈으로 감쌈. 빠르지만 — AdvisorServer는 순수 Java 라이프사이클(수동 ScheduledExecutorService·main())이라 Spring Boot의 빈/스케줄링/설정과 두 개의 라이프사이클이 충돌. 멀티테넌시(풀 N개)를 넣는 순간 싱글 풀 전제와도 안 맞음
  • 레시피 이식: server 모듈은 배포 안 하고, AdvisorServer가 보여준 배선 순서(복원→체크포인트 @Scheduled→셧다운 저장→감사 fan-out)를 Spring 구성으로 옮김. core+persistence만 의존
→ 레시피 이식. AdvisorServer는 "어떻게 배선하는지"의 문서로 완벽하지만, Spring 위에선 Spring의 라이프사이클로 다시 표현하는 게 자연스럽고 멀티테넌시로 가는 길도 열어둔다. gRPC 노출도 grpc-spring-boot-starter로 advisor.proto를 재사용.
결정 3고객 API 표면 — gRPC vs REST
  • Phase 1 = gRPC 그대로: advisor.proto가 이미 있고 buf로 가드됨. dogfood는 내가 클라이언트라 gRPC stub으로 충분. 새로 설계할 계약 0
  • REST 게이트웨이: 외부 고객 온보딩 마찰을 줄이지만(대부분 REST 기대), 계약을 새로 설계·유지해야 함 → 외부 고객이 실제로 생기는 Phase 3로 연기
  • 대시보드(Phase 2)는 웹이라 별도 — gRPC-web 또는 얇은 REST read 엔드포인트, 그때 결정
→ Phase 1은 gRPC, REST는 Phase 3. 지금 REST를 짓는 건 아직 없는 고객을 위한 투자. dogfood 검증이 먼저.
테넌시는 Phase 1에서 "결정 유보"가 정답: dogfood는 테넌트 1개(나)라 멀티테넌시 코드를 지금 짓지 않는다. 다만 방향은 "테넌트 = 풀 네임스페이스, 공유 프로세스"로 열어둔다(테넌트당 컨테이너는 운영이 무거움). 실제 격리 구현은 Phase 3.

§3Phase 1에 남은 실제 작업 — "뭐가 남았지"의 답

전부 호스팅/운영/판매 층. 엔진 로직은 0줄. 위에서 아래로 대략의 순서

  1. Spring Boot 프로젝트 스캐폴드 build
    Gradle(JDK 25), reputation-pool-core + reputation-pool-persistence 0.2.0 의존, grpc-spring-boot-starter. cloud 레포 루트에.
  2. ReputationPool을 Spring 빈으로 배선 (레시피 이식) core
    AdvisorServer 순서 그대로: 시작 시 PostgresResourceStore에서 복원 → @Scheduled 체크포인트 → PostgresAuditTrail 감사 fan-out(+보존 정책) → @PreDestroy 최종 저장. Clock 빈 주입.
  3. gRPC 서비스 구현 api
    advisor.proto의 6개 RPC를 Spring gRPC 서비스로 — 엔진 호출은 얇은 위임. (server 모듈의 ReputationAdvisorService가 참조 구현.)
  4. API key 인증 인터셉터 auth
    gRPC ServerInterceptor로 단일 키 검증(설정/env). 없거나 틀리면 UNAUTHENTICATED. 테넌트 식별의 자리(Phase 1은 상수 테넌트).
  5. Docker Compose 배포 ops
    앱 컨테이너 + PostgreSQL + Flyway 마이그레이션(persistence의 V1/V2 재사용). private 서버에 올릴 compose.yaml + 헬스체크.
  6. 운영 문서 + 시크릿/설정 ops
    API key·DB 자격증명·보존 기간(REPUTATION_POOL_AUDIT_RETENTION) 설정법, private 서버 배포 절차.
  7. Dogfood — 내 풀 실운영 검증
    실제 프록시/계정 풀을 Register로 등록하고 Acquire/Report 루프를 물림. 여기서 나오는 엔진 개선 요구는 공개 레포 이슈로.
과금·미터링·멀티테넌시·대시보드는 Phase 1에 없다. Phase 1의 목적은 매출이 아니라 "이 엔진이 실제 풀을 물고 몇 주 돌 때 무엇이 아픈가"를 몸으로 아는 것. 그 학습이 Phase 2·3의 우선순위를 정한다.

§4순서와 지금 할 일

순서무엇선행 조건누가
지금 이 결정 3개 확정 + cloud 레포에 Phase 1 이슈로 등록 — 승인 후 나
선행 0.2.0 릴리스에 persistence 배포 추가(결정 1의 귀결) #55 (릴리스 준비 PR 진행 중) 배포는 자격증명 머신
착수 §3 작업 1~2 (스캐폴드 + 엔진 배선) core+persistence 0.2.0 배포됨 에이전트
이어서 §3 작업 3~7 배선 동작 에이전트
결정 1이 릴리스에 주는 영향: persistence를 Phase 1에서 라이브러리로 쓰려면 0.2.0에 함께 배포하는 게 깔끔하다. 지금 진행 중인 #55 릴리스 준비 PR은 "core만" 전제로 쓰이고 있으므로, 이 결정을 승인하면 persistence publish 설정 추가를 릴리스 범위에 넣어야 한다(작은 build.gradle.kts 변경 + 좌표). 승인 여부를 알려주면 릴리스 PR에 반영하도록 지시하겠다.

REPUTATION-POOL-CLOUD · 프로덕션 마스터 플랜 · 2026-07-15

판매 가능한 프로덕션 SaaS로 —
한 번에 완성하고, 그다음 검증

목적 = 실제 판매할 SaaS. 전략 = dogfood 최소 MVP가 아니라 프로덕션 레벨까지 완성 후 검증(사용자 결정). 이 문서는 그 "완성"이 실제로 무엇을 포함하는지, 어떤 순서로 지을지, 그리고 바뀌지 않는 단 하나의 선행 게이트를 정리한다.

확정된 전략: 헤드리스 MVP로 굴려보며 배우는 대신, 백엔드·멀티테넌시·인증·과금·대시보드·관측·배포까지 프로덕션 품질로 지어 올린 뒤 검증한다. 아래는 그 완성본의 구성과 빌드 순서.

§0바뀌지 않는 단 하나의 게이트

먼저: 0.2.0 배포

전략이 무엇이든 cloud 백엔드는 reputation-pool-core + -persistence 0.2.0을 Maven Central에서 의존합니다. 그게 배포되기 전엔 cloud 코드가 컴파일조차 안 됩니다. 순서: ① 공개 레포 PR #59 머지 → ② 자격증명(GPG 키) 머신에서 publishToMavenCentral + v0.2.0 태그. 이 게이트가 프로덕션 빌드의 0번째 스텝입니다.

프로덕션까지 엔진을 실사용하게 되므로, 1.0.0 선언은 이 프로덕션 검증을 통과한 뒤로 미룹니다(앞선 버전 전략과 일치 — dogfood/검증 후 1.0.0).

§1"프로덕션 SaaS"가 실제로 포함하는 것

지금 cloud 이슈 #1~#7(dogfood MVP)로는 부족 — 판매 가능한 멀티테넌트 SaaS는 10개 도메인

D1 · 기존 #1~#2 확장
백엔드 코어 토대 있음

Spring Boot 셸 + 엔진 배선(AdvisorServer 레시피) + persistence. 어느 버전이든 필요한 토대.

D2 · 신규
멀티테넌시 판매 필수

테넌트 모델, 테넌트=풀 네임스페이스 격리, 테넌트별 스코프. 여러 고객을 한 서비스로 받으려면 필수.

D3 · #3 확장
API 표면: gRPC + REST gRPC 계약 있음

운영 데이터 플레인은 advisor.proto 재사용. 고객·대시보드용 REST + 컨트롤 플레인(테넌트·키·조회)은 신규 — advisor.proto엔 없음.

D4 · #4 확장
인증/인가 API key 자리 있음

테넌트별 API key 발급·회전·폐기, 대시보드 로그인(관리자). 싱글 키 → 키 관리 시스템으로.

D5 · 신규 (Phase 2였음)
대시보드 UI 완성도 핵심

프로덕션 프론트: 풀 상태·평판 곡선·이벤트 타임라인·테넌트/키 관리·사용량. 재료 = SubscribeEvents + D9 Metrics.

D6 · 신규 · 두 단계로 분리
과금/미터링 "판매"의 핵심

미터링(수집)은 테넌시 직후부터 — 사용량 데이터는 소급 수집 불가라 늦게 켜면 그만큼 영원히 비어 있음. 청구(Stripe)는 후반. 가격 모델(리스 수/풀 크기/좌석)은 결정 대기.

D7 · 신규
관측/SRE + OSS 연결

Metrics(코어 observability 포트 신설 → Micrometer), 모니터링·알림·SLO, rate limit. 포트는 공개 레포 이슈로.

D8 · 신규
배포/인프라 프로덕션

private 서버에 프로덕션급: TLS, 시크릿 관리, 스테이징/프로덕션 분리(compose 프로젝트 분리로 시작), DB 백업 + 복원 리허설(복원해본 적 없는 백업은 백업이 아님), 마이그레이션, CI/CD.

D9 · 신규
온보딩/문서/지원 고객 대면

가입 흐름, 고객용 API 문서·quickstart, 랜딩, 문의 채널·상태 페이지(최소형). 판매하려면 "어떻게 시작하고 어디에 물어보나"가 있어야.

D10 · 신규 · 셀프 리뷰에서 추가
법무/컴플라이언스 돈 받는 순간 필수

이용약관·개인정보처리방침, 결제 관련 법무(국내 판매 시 통신판매업 신고 등 사업 요건 확인), 데이터 보관·삭제 정책(감사 보존 정책과 연결). 코드가 아니라서 빠뜨리기 쉽지만 "판매"의 법적 실체.

핵심 설계 발견 ①: advisor.proto의 6개 RPC는 데이터 플레인(Register/Acquire/Report/Renew/Release/SubscribeEvents)입니다. 프로덕션 SaaS는 그 위에 컨트롤 플레인(테넌트 생성, API key 관리, 대시보드용 읽기 모델 — 풀 목록·리소스별 현재 평판/상태)이 추가로 필요하고, 이건 advisor.proto에 없습니다. D3의 상당 부분이 이 신규 컨트롤/조회 API 설계입니다.
핵심 설계 발견 ②: advisor.proto에는 테넌트 개념 자체가 없습니다 — Register/Acquire의 어떤 필드도 테넌트를 받지 않아요. 멀티테넌시는 wire 계약을 바꾸지 않고 API key → 테넌트 매핑을 인증 인터셉터에서 해석해 풀 네임스페이스로 라우팅하는 방식이어야 합니다 (proto를 바꾸면 공개 계약 breaking — buf 가드에 걸림). 이게 D2와 D4가 한 몸인 이유.

§2"한 번에" 안에서도 제정신인 빌드 순서

한꺼번에 프로덕션까지 가되, 통합이 서는지 확인하며 쌓는다 (빅뱅-후-검증의 통합 지옥 회피)

0. 0.2.0 배포 ────────────────── (게이트, 안 하면 아무것도 컴파일 안 됨) │ 1. D1 백엔드 코어 ─ 엔진이 gRPC 뒤에서 실제로 뜬다 (통합 증명) │ 2. D2 멀티테넌시 + D4 인증 ─ 테넌트/키 격리 └─ D6a 미터링 수집 ─ 여기서부터 사용량이 쌓인다 (소급 불가 — 늦으면 영원히 빈 데이터) │ 3. D3 REST + 컨트롤 플레인 ─ 대시보드가 부를 API + D7a Metrics 포트(OSS) │ 4. D5 대시보드 UI ─ 이제 부를 API가 다 있으니 그 위에 프론트 │ 5. D6b 청구(Stripe) + D7b 모니터링/SLO ─ 쌓인 미터 → 인보이스, 알림 │ 6. D8 배포 + D9 온보딩 + D10 법무 ─ 프로덕션 인프라 + 고객 진입 + 판매의 법적 실체 │ 7. 검증 (종료 기준 명시) ─ 아래 기준 전부 통과 → core 1.0.0 선언 ├─ 실제 풀 ≥ 1개를 스테이징 아닌 프로덕션에서 2주+ 무중단 운영 ├─ 대시보드만으로 일상 운영 가능 (SQL 직접 조회 없이) ├─ 테스트 테넌트 2개 이상에서 격리 검증 (교차 접근 불가 증명) ├─ 미터링→인보이스 왕복 1건 성공 (테스트 모드) └─ 백업 복원 리허설 1회 성공
솔직한 리스크 고지: 이건 혼자면 수개월 규모의 멀티테넌트 SaaS입니다(D2 격리·D6 Stripe·D5 프론트가 각각 큼). "빅뱅 후 한 번에 검증"의 고전적 위험은 통합 문제를 맨 끝에 발견하는 것 — 그래서 완성도를 낮추자는 게 아니라, 위 순서처럼 각 층이 서는지만 확인하며 프로덕션 품질로 쌓는 걸 권합니다. 층마다 "돈다"만 보고 다음으로.

착수 전에 못 박아야 할 결정 (프로덕션 v1이라 더는 못 미룸)

결정선택지기울기
테넌시 격리 수준 row-level(모든 테이블에 tenant_id) vs 스키마-per-테넌트 vs DB-per-테넌트 row-level 시작이 유력(운영 단순) — 단, persistence 스키마(V1/V2)에 tenant 축이 없어 cloud 쪽 스키마 확장 설계 필요. D2 첫 작업
대시보드 프론트 스택 React/Next.js vs 서버사이드(Thymeleaf 등) vs 기타 D5 착수 시 결정 — 실시간 이벤트(스트림)·차트가 요구사항이므로 그 기준으로
가격 모델 리스 횟수 종량 vs 풀 크기 티어 vs 좌석 D6a 미터링이 무엇을 셀지를 정하므로 D6a 전에 가설이라도 확정
백업 목표 일일 스냅샷 vs 연속(WAL) 아카이브 v1은 일일 + 복원 리허설로 시작, 유료 고객 붙으면 상향

§3공개 레포와의 연결 — Metrics 포트

D7(관측)의 핵심인 observability/metrics 포트는 공개 레포 core에 신설해야 합니다(AGENTS.md가 포트로 명시했으나 미구현). 이게 대시보드(D5)의 데이터 소스이자 모니터링(D7)의 소스라 — OSS 한 곳을 고치면 cloud 두 곳이 풀립니다. open-core 경계대로: 포트는 공개 레포(core), Micrometer 어댑터는 cloud(또는 공개 어댑터 모듈).

§4다음 단계 — 이슈/마일스톤 재구성

현재 cloud #1~#7은 "dogfood MVP" 프레이밍 → 프로덕션 프레이밍으로 갈아끼움

  • #1~#2 유지(백엔드 코어 = D1) — 프레이밍만 "dogfood" → "프로덕션 토대"로.
  • #3~#7 확장 + D2·D5·D6(a/b 분리)·D8·D9·D10 신규 이슈 — 위 10개 도메인을 이슈로.
  • 마일스톤 재편: MVP (dogfood) → 빌드 순서 §2의 단계별 마일스톤(예: v1-backend → v1-tenancy → v1-dashboard → v1-launch). 단일 거대 마일스톤보다 진행이 보임. 검증(§2-7 종료 기준)이 마지막.
  • 공개 레포: observability 포트 이슈 신설(core 0.3.0 마일스톤).
  • 위 결정 대기 4건(격리 수준·프론트 스택·가격 모델·백업 목표)은 해당 도메인 착수 시점에 각각 결정 — 격리 수준만은 D2가 첫 코드라 가장 먼저.
승인해주시면 이 재구성을 실제 이슈/마일스톤으로 반영합니다. 단, 착수 자체는 여전히 §0의 0.2.0 배포 후에 시작됩니다(백엔드가 그 아티팩트에 의존). 그 전까지는 계획·이슈 정비 + 배포 준비(PR #59 머지)가 할 수 있는 전부입니다.

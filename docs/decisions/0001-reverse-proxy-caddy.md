# ADR 0001 — 리버스 프록시로 Caddy 채택

- 상태: 채택됨 (accepted)
- 맥락: #15 프로덕션 인프라 (D8), 최초 도입은 #46
- 관련: `Caddyfile`, `compose.yaml`의 `caddy` 서비스

## 배경

대시보드(Next.js)와 API(Spring, `:8083`)가 별도 서비스로 나뉘어 있다. 브라우저가 둘을 서로 다른
오리진으로 접속하면 CORS가 필요하고 사용자 경험도 갈라진다. 그래서 **한 오리진 뒤로 묶는 리버스 프록시**가
필요하다: `/api`·`/actuator`는 백엔드로, 나머지는 대시보드로 라우팅해 same-origin으로 서빙한다
(dev의 `next rewrite`를 프로덕션에서 대체). 여기에 도메인·TLS 종료(#15 후속)까지 얹어야 한다.

현재 배포 형태는 **private 서버 + Docker Compose 단일 호스트**이며(PRODUCT.md의 "단순하게 시작, k8s는
필요해질 때"), 아직 수평 확장·멀티 인스턴스는 없다.

## 결정

리버스 프록시로 **Caddy**를 채택한다. 근거:

1. **자동 HTTPS.** 도메인만 지정하면 Let's Encrypt 인증서 발급·갱신을 Caddy가 자동 처리한다.
   Nginx/Apache는 certbot 등 별도 도구와 갱신 스케줄을 직접 붙여야 한다. 프로토타입~소규모 운영에서
   TLS 운영 부담이 사실상 0이다. (지금은 평문 `:80`, #15에서 도메인 붙일 때 `auto_https`를 켠다.)
2. **설정 단순성.** `Caddyfile`이 10줄 남짓이다. Nginx의 장황한 `server`/`location` 블록 대비
   학습·유지비가 낮아 작은 팀 운영에 유리하다.
3. **안전한 기본값 + 단일 바이너리.** HTTP/2·HTTP/3와 합리적 보안 디폴트가 기본 내장. 의존성 없는 Go
   정적 바이너리라 공식 `caddy:2` 이미지를 그대로 쓴다.
4. **요구가 "경로 라우팅 + 한 오리진" 수준.** upstream 두 개로 경로 분기 — 가장 적은 설정으로 되는 도구.

## 대안과 기각 사유

| 대안 | 강점 | 지금 기각한 이유 |
|---|---|---|
| **Nginx** | 성능·생태계, 세밀한 캐싱·레이트리밋·튜닝 | TLS 자동화가 별도(certbot), 설정 장황. 고성능·대규모 튜닝이 필요할 때 빛남 — 현 단계엔 오버스펙 |
| **Apache (httpd)** | 오래되고 기능 풍부 | 무겁고, 최신 리버스 프록시 프론트로는 잘 쓰지 않음. 이 용도엔 부적합 |
| **전용 LB (HAProxy / 클라우드 ALB·NLB·GCLB)** | L4/L7 로드밸런싱·헬스체크·수평확장 | 인스턴스가 1개라 분산할 대상이 없음. 클라우드 LB는 관리형이나 특정 클라우드에 종속 |

핵심은 로드밸런서의 본업(여러 인스턴스로 트래픽 분산)이 지금은 필요 없다는 것이다. 현재 필요한 것은
리버스 프록시(경로 라우팅 + 한 오리진 + TLS 종료)뿐이고, 거기에 TLS 자동화까지 얹어주는 Caddy가
비용 대비 최적이다.

## 결과

- 브라우저는 Caddy 한 오리진만 본다 → CORS 불필요. `Caddyfile`이 `/api`·`/actuator`→`app:8083`,
  나머지→`dashboard:3000`으로 라우팅.
- 모든 요청이 Caddy를 거치므로, 백엔드가 보는 소켓 IP는 항상 Caddy다. 실제 클라이언트 IP는
  `X-Forwarded-For`로 전달되며, 로그인 브루트포스 방어(#28)가 이를 스로틀 기준으로 쓴다. 이때
  **신뢰 경계는 네트워크**다 — `:8083`은 Caddy만 접근 가능해야 하고 앱을 외부에 직접 노출하면 안 된다
  (그러지 않으면 `X-Forwarded-For` 위조로 스로틀 우회가 가능). 자세한 내용은 `security.md` 참고.

## 재검토 트리거

다음 중 하나가 오면 이 결정을 다시 본다:

- 트래픽 증가로 **멀티 인스턴스 수평 확장**이 필요해질 때 → Caddy 앞에 전용/클라우드 LB를 두거나 구성 변경.
- 세밀한 캐싱·레이트리밋·WAF·성능 튜닝이 절실해질 때 → Nginx 등으로 교체 검토.

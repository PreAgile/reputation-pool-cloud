§3설계 갭 — "평판 곡선"은 지금 어디에도 없다

이 스펙을 데이터에서 역산하다 발견한 유일한 공급 불가 항목

score 히스토리는 현재 아무 데도 저장되지 않습니다. 스냅샷(V1)은 현재 score만 갱신하고, 감사 이벤트(V2)에는 score 컬럼 자체가 없습니다(방금 스키마로 확인). 즉 화면 2의 "평판 곡선(시간에 따른 score)"은 지금 데이터로는 그릴 수 없습니다 — 상태 변화 마커(냉각/회복 시점)까지만 가능.
해결 옵션내용판정
cloud 쪽 주기 샘플링체크포인트 루프에 편승해 테넌트별 score 시계열 테이블에 주기 기록(예: 1분). core·공개 스키마 무변경채택 — #10(미터링) 또는 #11(읽기 모델)에 편입, open-core 경계 유지
감사 이벤트에 score 추가공개 레포 V2 스키마·이벤트 확장기각 — 감사는 "무슨 일이"의 장부지 시계열 저장소가 아님, 공개 계약 변경
observability 포트(#60) 경유메트릭 백엔드(Prometheus류)에 score 게이지 → 대시보드가 그걸 조회보완 — 모니터링(#14)엔 이 경로가 맞지만, 고객 화면이 메트릭 스토어에 의존하는 건 결합 과다. v1은 샘플링 테이블

도메인 후보: reputationpool.io / reputationpool.dev 계열 — OSS 이름과 일치가 신뢰 자산(구매는 직접, 이후 DNS→private 서버, Caddy로 TLS 자동화 = #15). 전부 한 도메인의 서브도메인로 묶어 인증서·쿠키 관리 단순화.

이 제품 고유의 마케팅 자산 = 오픈소스 엔진. Supabase·Grafana가 증명한 open-core 판매 공식: "엔진은 공개돼 있고(GitHub 링크), Lincheck으로 선형성이 증명돼 있으며, mutation testing으로 테스트의 이빨까지 검증했다" — 아직 고객 로고가 없는 신생 SaaS에서 소셜 프루프를 대신하는 신뢰 신호가 정확히 이것. 랜딩의 히어로가 이걸 팔아야 한다.
